Мета роботи
Метою роботи є дослідження та систематизація основних принципів написання якісного, чистого та узгодженого PHP-коду відповідно до стандартів PSR-1 «Basic Coding Standard» та PSR-12 «Extended Coding Style Guide», а також демонстрація того, як дотримання цих стандартів спрощує рефакторинг і супровід програмних проєктів.
Хід роботи (Слайд 2)
Робота виконувалася послідовно згідно з поставленими завданнями, на основі матеріалів презентації з теми «Важливість правил оформлення коду (PSR-1, PSR-12)»:
Вибір і обґрунтування теми (Слайд 2)
Було обрано тему «Важливість правил оформлення коду (PSR-1, PSR-12)» для мови програмування PHP. Стандарти PSR розробляються організацією PHP-FIG (PHP Framework Interop Group) і призначені для уніфікації підходів до написання коду в різних командах і фреймворках. 
PSR-1 визначає базові вимоги до PHP-файлів, іменування класів, методів та констант, а PSR-12 деталізує правила форматування: відступи, довжина рядків, розташування дужок, пробіли, порядок оголошень у файлах тощо. 
Іменування змінних (Слайд 3)
За рекомендаціями PSR-12 змінні, властивості класів і параметри функцій у PHP мають використовувати стиль camelCase. Такий підхід забезпечує однорідність і дозволяє швидко відрізнити змінні від класів та констант. 
Основні правила:
• назви мають бути описовими й зрозумілими;
• скорочення та односимвольні імена (крім лічильників циклу) слід уникати;
• не можна змішувати різні стилі на кшталт $User_name, $USERName тощо.
Поганий приклад:
$usr;
$User_name;
$A;
   Гарний приклад:
$userName;
$maxItemsCount;
$orderTotalPrice;
У правильних прикладах з назви одразу зрозуміло, за що відповідає змінна, без додаткових коментарів чи аналізу контексту.

Іменування класів і структур (Слайд 4)
За PSR-1 та PSR-12 назви класів повинні використовувати стиль PascalCase (StudlyCaps), де кожне слово в назві починається з великої літери. 
    Основні рекомендації:
• один клас — в окремому файлі;
• ім’я класу має відповідати імені файлу;
• простір імен відображає структуру директорій.
    Поганий приклад:
class user_manager { ... }
class usermanager { ... }
    Гарний приклад:
class UserManager { ... }
class OrderRepository { ... }
Такий підхід полегшує автозавантаження через PSR-4, робить структуру проєкту передбачуваною, а пошук потрібних класів — очевидним.

Іменування констант (Слайд 5)
Для констант у PHP традиційно використовується стиль UPPER_CASE з підкресленнями для багатослівних назв. Це чітко сигналізує, що значення є незмінним протягом роботи програми. 
Правила:
• усі літери — великі;
• слова розділяються підкресленням;
• назва відображає смисл значення.
Поганий приклад:
const maxspeed = 100;
const MaxSpeed = 100;
Гарний приклад:
const MAX_SPEED = 100;
const DEFAULT_TIMEOUT_SECONDS = 30;
У класах можуть використовуватись і class constants, що дозволяє групувати важливі налаштування та значення безпосередньо там, де вони застосовуються.
Коментарі (Слайд 6)
Стандарти PSR не нав’язують суворого формату коментарів, але рекомендують пояснювати «чому», а не просто дублювати те, що й так видно з коду. 
Коричневый_и_Кремовый_Деликатны…
Добрі коментарі:
• пояснюють причину рішення чи особливість реалізації;
• описують обмеження та нетривіальну логіку;
• не повторюють очевидні дії.
Поганий приклад:
// Збільшуємо лічильник на 1
$counter++;
Гарний приклад:
// Лічильник використовується для обмеження кількості невдалих спроб входу
$failedLoginAttempts++;
Короткі, змістовні коментарі, доповнені PHPDoc для методів і класів, суттєво полегшують супровід коду новими членами команди.
Структура файлу та простори імен (Слайд 7)
PSR-12 задає чіткий порядок елементів у PHP-файлі. Це допомагає уникати хаосу, коли оголошення змішані з логікою та підключеннями. 
Рекомендована структура:
1. Відкривальний тег <?php та декларація declare(strict_types=1); (за потреби).
2. Оголошення namespace.
3. Блок use для підключення залежностей.
4. Оголошення класів, інтерфейсів, трейтів, функцій.
Кожен файл повинен:
• містити або оголошення сутностей (класів, функцій тощо), або побічні ефекти (вивід, зміна налаштувань), але не обидва одночасно (вимога PSR-1); 
• використовувати кодування UTF-8 без BOM;
• не містити закривального тега ?> у файлах, де є лише PHP-код.
Чітка структура файлів спрощує навігацію, рефакторинг і зменшує кількість помилок, пов’язаних з некоректним підключенням або повторними оголошеннями.
Використання констант у PHP та уникнення «магічних чисел» (Слайд 8)
«Магічні числа» — це числові літерали в коді, сенс яких неочевидний. Їхнє використання ускладнює підтримку та зміну логіки.
PSR-стандарти рекомендують:
• замінювати такі значення іменованими константами (const, private const);
• зберігати важливі налаштування в одному місці, а не розкидати по коду. 
Коричневый_и_Кремовый_Деликатны…
Поганий приклад:
if ($speed > 120) {
    // ...
}
Гарний приклад:
class Car
{
    private const MAX_ALLOWED_SPEED = 120;

    public function accelerate(int $speed): void
    {
        if ($speed > self::MAX_ALLOWED_SPEED) {
            // ...
        }
    }
}
У другому випадку значення 120 має чітке ім’я, тому зміни обмеження швидкості відбуватимуться в одному місці, без небезпеки пропустити ще якесь «захардкожене» число.
Основні конвенції PSR-1 та PSR-12 (Слайд 9)
Під час роботи були виділені ключові правила, які безпосередньо впливають на якість PHP-коду:
Іменування:
• змінні, параметри, методи — camelCase;
• класи, інтерфейси, трейти — PascalCase;
• константи — UPPER_CASE з підкресленнями. 
Форматування:
• відступи — 4 пробіли, без табуляцій;
• максимальна довжина рядка — до 120 символів (рекомендовано);
• фігурні дужки для класів та методів — з нового рядка;
• пробіли після ключових слів if, for, while тощо;
• логічно пов’язані блоки коду відокремлюються порожніми рядками.
Структура:
• один клас — один файл;
• послідовний порядок namespace, use та оголошень;
• уникнення require/include у довільних місцях бізнес-логіки.
Дотримання цих правил дозволяє налаштувати автоматичне форматування коду за допомогою інструментів-лінтерів і форматерів (наприклад, phpcs, phpcbf), що ще більше знижує кількість стилістичних помилок.
Що таке рефакторинг у контексті PSR (Слайд 10–11)
Рефакторинг — це процес покращення внутрішньої структури коду без зміни його зовнішньої поведінки. У контексті PSR це означає:
• перейменування змінних, класів та методів згідно зі стандартами;
• винесення «магічних чисел» у константи;
• упорядкування структури файлів та просторів імен;
• розбиття довгих функцій на менші, з чітко визначеною відповідальністю.
Прикладом може бути перехід від хаотично оформленого PHP-файлу до варіанту, що чітко відповідає PSR-1 та PSR-12: зі структурованими класами, осмисленими іменами й акуратними відступами. Такий код:
• легше тестувати;
• простіше читати та пояснювати;
• зручніше розширювати новим функціоналом.
Висновки: як PSR покращує якість PHP-коду (Слайд 12)
У результаті виконання роботи було досягнуто поставленої мети — систематизовано основні вимоги стандартів PSR-1 та PSR-12 для мови PHP і продемонстровано їхній вплив на якість коду.
Дотримання PSR-стандартів забезпечує:
1. Читабельність і зрозумілість.
Єдиний стиль оформлення спрощує сприйняття логіки програми й зменшує когнітивне навантаження на розробника.
2. Простоту підтримки й розширення.
Структурований код легше змінювати, налагоджувати та доповнювати новими функціями.
3. Зменшення кількості помилок.
Уникнення хаотичного форматування, «магічних чисел» і неочевидних рішень знижує ризик логічних та синтаксичних помилок.
4. Кращу командну взаємодію.
Стандартизований код дає змогу різним розробникам швидко включатися в проєкт і розуміти чужі зміни без довгих пояснень.
5. Професійний вигляд коду.
Проєкти, що дотримуються PSR-стандартів, виглядають охайно й викликають більшу довіру з боку замовників і колег.
Отже, впровадження PSR-1 та PSR-12 у повсякденну практику PHP-розробника є важливим кроком до підвищення якості програмного забезпечення та ефективної командної роботи.
Використані джерела
1. PHP-FIG. PSR-1: Basic Coding Standard.
Офіційна специфікація стандарту PSR-1, що визначає базові вимоги до PHP-файлів, іменування класів, методів і констант.
Доступно за посиланням:
https://www.php-fig.org/psr/psr-1/ php-fig.org+1
2. PHP-FIG. PSR-12: Extended Coding Style Guide.
Розширений гайд зі стилю коду, який доповнює PSR-1 та замінює застарілий PSR-2.
Доступно за посиланням:
https://www.php-fig.org/psr/psr-12/ php-fig.org+1
3. PHP-FIG — PHP Framework Interop Group.
Офіційний сайт організації, що розробляє PSR-стандарти для PHP.
Доступно за посиланням:
https://www.php-fig.org/ php-fig.org+1
4. PHP Manual — Language Reference, Basic syntax.
Офіційна документація PHP, розділ про базовий синтаксис, теги PHP та загальні правила написання коду.
Доступно за посиланням:
https://www.php.net/manual/en/language.basic-syntax.php php.net+1
5. Hexlet. «Линтер. Основы PHP» (курс «PHP-basics»).
Матеріал про роль лінтингу та використання PSR-12 як стандарту стилю коду в сучасних PHP-проєктах.
Доступно за посиланням:
https://ru.hexlet.io/courses/php-basics/lessons/linting/theory_unit ru.hexlet.io
